# 추천 시스템 구현 정리

## 원칙

- **장소 단위로 순위를 매긴다.**  
  리뷰 단위가 아니라, 장소별로 점수를 모아서 그 장소의 “추천 순위”를 정한다.

---

## 1. 전체 흐름

```
사용자 자연어
    → Backend: 카테고리 + 위치 추출 (OpenAI 등)
    → Backend: Kakao Map으로 위·경도 조회
    → GgUd-AI: 위경도 반경 후보 + 카테고리별 벡터 검색 → 장소별 점수 계산
    → GgUd-AI: place_id, ai_score, distance_from_midpoint 만 반환
    → Backend: S3(버킷1)에서 메타데이터 조회 후 클라이언트에 응답
```

---

## 2. GgUd-AI 내부 동작

### 2.1 입력

- **요청**: `query`(자연어), 선택적으로 `latitude`, `longitude`, `limit`
- **쿼리에서 추출**: 카테고리 4종(동행자/메뉴/분위기/목적) + 위치 문자열  
  → 위치는 Backend가 Kakao Map으로 위·경도 변환 후 넘기거나, GgUd-AI가 추출한 위치로 보조 사용

### 2.2 단계별 처리

| 순서 | 단계 | 설명 |
|------|------|------|
| 1 | **위치 필터(선택)** | 위도·경도·반경(km)이 있으면, 해당 반경 안의 장소 ID만 후보로 사용. 반경 없으면 전 장소가 후보. |
| 2 | **카테고리만 검색** | 요청에서 추출된 카테고리만 사용. 없는 카테고리는 검색하지 않음. |
| 3 | **카테고리별 벡터 검색** | 각 카테고리 값(문자열)을 임베딩한 뒤, **후보 장소 안에서만** 해당 카테고리의 `review_embeddings` **각 행(리뷰)**에 대해 쿼리 벡터와의 코사인 거리를 구함. |
| 4 | **장소별 점수** | 같은 장소에 속한 **리뷰별 거리의 평균(avg)**을 내서 그 장소의 유사도로 씀. 그 유사도에 카테고리 가중치(메뉴 0.4, 동행자/분위기/목적 각 0.2)를 곱해 합산. 중복 제거나 빈도수 보정 없음. |
| 5 | **순위** | 장소별 합산 점수로 정렬해 상위 N개 선택. |

즉, **“장소별로 순위 매기기”** = 장소 단위로 리뷰들의 유사도 평균을 모아서, 그 장소의 최종 점수로 쓰는 방식이다.

### 2.3 출력

- **place_id**: 장소 PK  
- **ai_score**: 0~100 절대 점수 (카테고리 가중치 합 대비 비율로 환산)  
- **distance_from_midpoint**: (선택) 중간지점으로부터의 거리(km). 미구현 시 null.  
- 메타데이터(이름, 주소, 위경도 등)는 **GgUd-AI가 주지 않고**, Backend가 S3 버킷1에서 조회해 채운다.

---

## 3. 데이터 구조 (RDS)

### 3.1 임베딩 저장

- **테이블**: `review_embeddings`  
- **단위**: 리뷰 하나당, 카테고리별로 추출된 “값”마다 한 행.  
  같은 문장("빵이 맛있어요")이라도 **리뷰가 다르면 별도 행**.
- **유니크**: `(place_id, review_id, category, value_text)`
- **컬럼**: `place_id`, `review_id`, `category`, `value_text`, `embedding`

### 3.2 점수 계산 시

- 카테고리별로 해당 `category` 행만 조회.
- **각 행(리뷰)**에 대해 쿼리 벡터와의 코사인 거리를 구하고, **장소별로 그 거리의 평균(avg)**을 낸 뒤 유사도로 변환.
- 중복 제거·빈도수 보정 없이, “리뷰별 유사도 검색 → 장소별 평균으로 랭킹”만 수행.

---

## 4. 설정

- **반경 기본값**: `RECOMMENDATION_DEFAULT_RADIUS_KM` (기본 10.0 km)  
- **상위 개수**: `RECOMMENDATION_TOP_K` 또는 요청의 `limit`  
- **카테고리 가중치**:  
  - 메뉴(menu): 0.4  
  - 동행자(companion): 0.2  
  - 분위기(mood): 0.2  
  - 목적(purpose): 0.2  

---

## 5. 임베딩 → 순위 매기기 (상세)

**저장 단계 (사전)**  
- 리뷰 텍스트에서 LLM이 카테고리(동행자/메뉴/분위기/목적) 값을 추출한다.  
- 각 카테고리별로 나온 값 문자열(예: menu="파스타", companion="연인")마다 OpenAI로 **한 번씩 임베딩**해서 `review_embeddings`에 저장한다.  
- 한 행 = (장소, 리뷰, 카테고리, 값 텍스트, 1536차원 벡터). 같은 문장이라도 리뷰가 다르면 별도 행.

**추천 요청 시**  
1. **쿼리 해석**  
   - 사용자 자연어에서 카테고리 값들(예: menu="디저트", companion="친구")과 위치를 추출한다.  
   - 위치가 있으면(또는 Backend가 넘긴 위경도가 있으면) 그 점 기준 반경(km) 안의 장소 ID만 **후보**로 둔다.

2. **카테고리별 벡터 검색**  
   - 추출된 카테고리만 쓴다. 값이 비어 있으면 그 카테고리는 건너뛴다.  
   - 각 카테고리마다:
     - 해당 카테고리 값 문자열을 **같은 모델로 임베딩**해 쿼리 벡터를 만든다.  
     - DB에서는 `review_embeddings` 중 `category = 해당 카테고리`이고 `place_id IN 후보`인 행만 본다.  
     - 각 행의 `embedding`과 쿼리 벡터의 **코사인 거리**를 구한다. (pgvector 기준, 거리 0에 가까울수록 유사)

3. **장소별 점수 (같은 카테고리 안에서)**  
   - 각 **리뷰(행)**에 대해 쿼리 벡터와 그 행의 임베딩의 코사인 거리를 구함.  
   - 같은 `place_id`를 가진 행들에 대한 **거리의 평균(avg)**을 그 장소의 해당 카테고리 유사도로 씀.  
   - 평균 거리를 유사도 점수로 바꾸기 위해 `similarity = 1.0 - (distance / 2.0)` 사용.  
   - **중복 처리나 빈도수 보정 없음**: 리뷰1·리뷰2 둘 다 메뉴=파스타면 두 행 모두 저장하고, 검색 시에도 두 거리 각각 구한 뒤 평균만 냄.

4. **카테고리 합산 (장소별)**  
   - 장소 A에 대해:  
     - (menu 유사도 × 0.4) + (companion × 0.2) + (mood × 0.2) + (purpose × 0.2) = **raw_score(A)**  
   - 유사도는 위에서 구한 **리뷰별 거리의 평균**을 `1 - distance/2` 로 변환한 값.

5. **순위**  
   - 모든 후보 장소를 **raw_score 내림차순**으로 정렬하고, 상위 N개(limit)만 골라서 반환한다.  
   - API에서는 이 raw_score를 가중치 합(0.4+0.2+0.2+0.2=1.0) 기준으로 0~100으로 환산해 `ai_score`로 준다.

---

## 6. 예외·특수 케이스

- **추출된 카테고리가 하나도 없을 때**  
  - 위치 필터가 있으면: 반경 안의 장소를 **거리순**으로만 반환하고, `ai_score`는 0으로 둠.  
  - 위치도 없으면: 빈 목록 반환.
- **후보 장소가 0개일 때**  
  - 빈 목록 반환.

---

## 7. 정리

| 항목 | 내용 |
|------|------|
| **순위 단위** | 장소별 |
| **점수 산출** | 카테고리별 **리뷰(행) 단위 유사도(거리)**를 구한 뒤, 장소별 **평균(avg)** → 가중 합산 |
| **검색 범위** | 위치가 주어지면 먼저 위경도 반경으로 후보 축소 → 그 안에서만 벡터 검색 |
| **검색 대상** | 요청에 있는 카테고리만 사용 |
| **API 응답** | place_id, ai_score, distance_from_midpoint만 반환, 메타데이터는 Backend S3 조회 |

이 문서는 현재 구현과 일치하도록 작성되었으며, “장소별로 순위 매기기”를 기준으로 한 추천 시스템 설계를 정리한 것이다.
